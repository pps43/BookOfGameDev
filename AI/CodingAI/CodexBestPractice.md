# Codex Best Practice

## 常见任务
- 解释核心逻辑、模块关系、数据流向、调用链
- 优化现有代码可读性可维护性
- 生成模板代码/外围脚手架代码、单元测试
- 根据BUG描述先行诊断，甚至直接尝试修复

## 人为控制任务复杂度

- **单次任务控制在「约 1 小时人类工作量 / 几百行代码改动」**。不急于提高任务量。
- 大任务先拆成可并行子任务：`探索`、`实现`、`验证`。
- 每个任务只定义一个可验收结果，避免“顺手再做一点”。

## 复杂任务时的工作流

1. `Ask/Plan` 先产出实现计划与风险点。
2. `Code` 执行最小可行改动。
3. 立刻跑验证（测试、构建、关键路径人工测试）。
4. 失败时先修环境和约束，再让模型重试。

## 提示词依然非常重要

- 提供约束：性能上限、兼容范围、禁止改动区域、代码风格。通用的建议放到`AGENTS.md`。
- **提供例子：这在2026.4依然很有用**（我看了OpenAI的员工现场演示）
- 如果可以，尽量提供“参考实现在哪”：`按 X 模块的方式实现 Y`。
- 对输出做要求：`仅改哪些文件`、`需要哪些测试`、`验收标准`。
- 让 Codex 主动补边界测试：空输入、极值、异常路径、并发场景。
- Codex擅长重构。但对跨文件重构，要求输出影响面摘要（模块、调用方、潜在破坏点）。

## 保护你的心流

- 将临时想法和低优先级修复交给AI，避免打断你当前正在思考的事。
- **在 agent 思考或执行期间，你可以继续补充上下文，不必等待它结束上一个任务**。


## 创建和使用 skill

- 一个 skill 只解决一类高频问题，避免“大而全”。
- 对复杂需求用“skill 串联”而不是单 skill 膨胀。**skill中可以调用其他skill**。
- 先在小任务验证 skill，再推广到复杂任务；每次失败都在skill中记录失败模式。
- skill 文档写成固定结构：`适用场景`、`输入样例`、`执行步骤`、`输出样例`、`反例`等等。
- 将 skill 与 `AGENTS.md` 分工：`AGENTS.md` 管仓库通用规则，skill 管专项流程。
